# テストケース仕様書（正式版）

両システム（指導報告・指導時間確認票 / イスト勤怠レポート for EMPS）の全ロール・全操作を対象に、
バグ・不具合・改善点を洗い出し、客先導入可能な品質へ仕上げるためのテスト仕様。

## 0. 本書の位置づけとフェーズ計画

| フェーズ | 内容 | 状態 |
|---|---|---|
| **Phase 1** | 共通＋両システムの**優先ケース**（①主要フロー〔最終承認まで〕②権限境界 ③直近改修の回帰）＋代表的な主要異常系 | **本書で完成（今回）** |
| Phase 2 | 異常系・境界値・UI/レスポンシブ・出力(PDF/CSV/Excel)・通知メールの網羅展開 | 次回 |
| Phase 3 | 自動テスト（Playwright e2e / pytest）の実装・実行、手動チェック実施、**バグ一覧の作成** | 次回（※フロー系E2Eは**メール実送信の無効化が前提**。§5参照） |
| Phase 4 | 検出バグの改修（個別の改修タスク） | 次回 |

> 実施方法は **併用**（自動 e2e/pytest ＋ 手動チェックリスト）。§4.5 参照。

## 1. 目的とゴール
両システムを「人間が実際に行う操作パターン」で網羅的にテストし、不具合を洗い出して客先導入可能な品質に仕上げる。
成果物：本仕様書（テストケース）／テスト実施結果（合否・バグ一覧）／バグ修正。

## 2. 対象システムとロール対応

| 区分 | 製品名 | コード | ポート | ログイン |
|---|---|---|---|---|
| 既存 | 指導報告・指導時間確認票 | `backend/` | 8000 | フォーム送信 |
| 新 | イスト勤怠レポート for EMPS | `new_backend/` | 8001 | JSON（複数ロール時はロール選択） |

| フロー位置 | 既存 | 新(EMPS) |
|---|---|---|
| 作成者 | 講師 | 講師 |
| 1次承認（外部） | 保護者 | 学校 |
| 事務処理・受領 | 受付担当(admin_receiver) | 事務(office) |
| 最終承認 | 再鑑(admin_reviewer) | 営業(sales) |
| フロー外（管理） | 管理者(admin_master)・管理責任者(admin_chief) | 経理(admin_master)・管理責任者(admin_chief) |

承認・差戻しは両系とも**参照画面(report_view)**に集約済み。新の経理は契約管理（フロー外）、管理責任者は学校確認スキップ可。

## 3. 業務フロー

**既存**：`draft →[講師:提出]→ awaiting_parent_approval →[保護者:承認]→ parent_approved → submitted_to_admin →[受付:受領]→ received →[再鑑:最終承認]→ admin_approved`
- 差戻し（コメント必須）→ returned_to_tutor → 講師が再提出。完了後 →[再鑑]→ returned_to_receiver。先月以前の未処理 → 管理者がクローズ。skip_parent_approval=ON は保護者承認を飛ばす。

**新(EMPS)**：`draft →[講師:学校へ依頼]→ awaiting_school →[学校:承認]→ awaiting_office →[事務:承認]→ awaiting_sales →[営業:承認]→ approved`
- 差戻し：学校/事務→returned_to_tutor、営業/完了後→returned_to_office。月分超過時は提出時に awaiting_office_precheck（事務事前確認）を前段挿入。管理責任者は skip_school で学校確認省略。

## 4. テスト体系

### 4.1 観点カテゴリ
正＝正常系／異＝異常系／権＝権限・セキュリティ／境＝境界値・データ／Ｕ＝UI・操作性／出＝出力・通知／整＝整合性・並行

### 4.2 ID採番
`{SYS}-{ROLE}-{連番}`：SYS=C共通/L既存/N新。ROLE=TUT講師/PAR保護者/SCH学校/REC受付/OFF事務/REV再鑑/SAL営業/MAS経理・管理者/CHF管理責任者/CMN共通/OUT出力/SEC権限横断。

### 4.3 優先度
A=必須（主要フロー・最終承認・重大権限・回帰）／B=重要／C=周辺。

### 4.4 1ケースの記載項目
ID／タイトル［観点・優先度・実施方法］／前提／手順／期待結果／（実施結果・判定は実施時に記入）

### 4.5 自動／手動の振り分け
| 実施方法 | 対象 |
|---|---|
| 自動 e2e（Playwright・実画面） | ログイン、遷移、権限境界(URL直打ち)、承認フロー、差戻し・再提出、必須/重複バリデーション、ボタン活殺 |
| 自動 pytest（API・サービス層） | ワークフロー全遷移、ロール権限、削除/無効ガード、No採番、エクスポート生成(列幅・行数・NULL除外)、メールキュー |
| 手動（チェックリスト＋スクショ） | PDF/帳票の目視レイアウト、PC/スマホのレスポンシブ、メール文面、印刷、操作感 |

## 5. テスト環境・データ

- 環境：dev（`docker compose up -d`、http://localhost:8000 / 8001、MailHog http://localhost:8025）。
- ⚠️ **メール安全（最重要・厳守）**：現在の dev `.env` は `MAIL_BACKEND=smtp`（実Gmail送信 `smtp.gmail.com` / 送信元 `tutor.reports.system@gmail.com`）。**提出・承認・差戻し・招待など全フロー操作は実メールを送信する**（バースト送出のリスク）。
  - **フロー系のE2E/手動テストを実施する前に、必ず実送信を無効化する**：`MAIL_BACKEND=console`（送らずログのみ）または MailHog 経由（`SMTP_HOST=mailhog` / `SMTP_PORT=1025` / `SMTP_TLS=none`）へ切替し、`docker compose up -d` で反映。**切替が確認できるまでフロー系の実行は保留**。
  - pytest（API/サービス層）は `conftest.py` が `MAIL_BACKEND=console` を強制するため、実送信なしで安全（いつでも実行可）。
  - ログイン・参照・直接seed（`seed.py` 等のユーザー直接作成）は送信契機がないため安全。
- **QA用seed（Phase 3で用意・冪等）**：
  - 既存：講師(L)・保護者・受付・再鑑・管理者・管理責任者 各1＋担当(講師×保護者×生徒)＋当月の指導報告（記入あり/空行混在）。
  - 新：講師・学校・事務・営業・経理・管理責任者 各1（既存 `create_test_users` の office1/sales1/school1 を踏襲）＋経理が登録した契約＋担当＋業務連絡表（勤務/有給/欠勤・空行混在）。
- パスワードは全アカウント `Passw0rd!`。

## 6. テストケース（Phase 1）

### 6.1 共通（C）

**C-CMN-001｜ログイン成功［正・A・自動e2e］**
- 前提：有効アカウント（初回変更済み）。
- 手順：1) `/login` 2) メール・パスワード入力→ログイン。
- 期待：ロール別トップへ遷移（講師=報告書一覧／保護者=承認管理／運営=ダッシュボード等）。ヘッダー/サイドバーに製品名・氏名・ロール表示。

**C-CMN-002｜ログイン失敗（誤パスワード）［異・A・自動e2e］**
- 手順：正しいメール＋誤パスワードでログイン。
- 期待：エラー表示、遷移しない、トークン未発行。

**C-CMN-003｜存在しないユーザー／空入力［異・B・自動e2e］**
- 期待：汎用エラー（ユーザー有無を区別しない）。必須未入力はフォーム検証で送信不可。

**C-CMN-004｜未所属システムへのログイン拒否（新）［権・A・自動e2e/pytest］**
- 前提：`allowed_systems` に `new` を含まないユーザー。
- 手順：8001 の `/login` でログイン。
- 期待：403＋案内文、ログイン不可。（逆：legacy専用ユーザーが 8000 はログイン可）

**C-CMN-005｜未ログインで保護ページにアクセス［権・A・自動e2e］**
- 手順：未ログインで `/tutor/reports`、`/admin/users` 等へ直接アクセス。
- 期待：ログイン画面へリダイレクト（またはAPIは401）。

**C-CMN-006｜ログアウト［正・A・自動e2e］**
- 期待：トークン破棄、保護ページに戻れない。

**C-CMN-007｜初回ログイン時パスワード変更必須［正・B・自動e2e］**
- 前提：初期パスワード＋`must_change_password=true`。
- 期待：自動で `/change-password`、変更前は他画面不可、変更後は通常トップ、次回以降は要求されない。

**C-CMN-008｜招待リンクからのアカウント登録［正・B・自動e2e/手動］**
- 前提：有効な招待トークン。
- 手順：`/register?token=...` で氏名・パスワード設定。
- 期待：登録完了→ログイン可。講師は講師No、運営はロールが表示。

**C-CMN-009｜期限切れ/無効な招待リンク［異・B・自動e2e］**
- 期待：「招待リンクを確認できません」を表示し登録不可。

**C-CMN-010｜パスワードリセット申請→再設定［正・B・自動e2e/手動］**
- 手順：`/forgot-password`→メール送信→リンクから新パスワード設定。
- 期待：メール（MailHog）受信、再設定後にログイン可、旧パスワードは不可。

### 6.2 既存システム（指導報告・指導時間確認票）

#### 講師（TUT）
**L-TUT-001｜指導報告の作成（明細入力→保存）［正・A・自動e2e］**
- 前提：当月、担当（講師-保護者-生徒）紐付け済み、当月未作成。
- 手順：報告書一覧 →「指導報告作成」で生徒・表示月選択 → 指導日/開始/終了/休憩/科目/内容入力 → 保存。
- 期待：一覧に1行追加、合計指導時間に加算、ステータス=下書き。

**L-TUT-002｜同一指導日の重複登録ガード［異・A・自動e2e］**
- 手順：既存と同じ指導日で登録。
- 期待：重複エラー表示、登録不可（API側でも拒否）。

**L-TUT-003｜前回の記入分をコピー［正・B・自動e2e］**
- 期待：直近記入行の内容（日付以外）が次行へコピーされる。

**L-TUT-004｜保護者へ承認依頼（提出）［正・A・自動e2e/pytest］**
- 前提：下書きが1件以上、skip_parent_approval=OFF。
- 手順：対象報告で「提出（保護者へ依頼）」。
- 期待：→ awaiting_parent_approval。保護者へ承認依頼メール。監査ログ記録。講師側は編集不可。

**L-TUT-005｜skip_parent_approval=ON 時の提出［正・B・pytest］**
- 期待：保護者承認を飛ばし submitted_to_admin（運営）へ直接遷移。

**L-TUT-006｜差戻し後の修正・再依頼［正・A・自動e2e/pytest］**
- 前提：returned_to_tutor の報告（差戻しコメントあり）。
- 手順：承認管理で差戻し内容を確認→修正→再依頼。
- 期待：差戻しコメントが見える、修正保存後 awaiting_parent_approval へ戻る。

**L-TUT-007｜他講師の報告書は見えない／操作不可［権・A・自動e2e/pytest］**
- 期待：自分の担当のみ。他人の報告書IDへの直接アクセスは拒否。

- 手順：承認管理→参照画面で承認。

**L-PAR-002｜差戻し（コメント必須）［異・A・自動e2e/pytest］**
- 手順：コメント未入力で差戻し→次にコメント入力で差戻し。
- 期待：未入力は不可（検証）。入力後 returned_to_tutor、講師へ通知（メール＋画面バッジ）。

**L-PAR-003｜他人の子の報告書は不可［権・A・自動e2e/pytest］**
- 期待：一覧に出ない、URL直打ち・承認APIとも403。

#### 受付担当（REC）
**L-REC-001｜受領［正・A・自動e2e/pytest］**
- 前提：submitted_to_admin。
- 手順：受付ログイン→参照画面で受領。
- 期待：→ received。

**L-REC-002｜受付による差戻し［正・B・pytest］**
- 期待：コメント必須で returned_to_tutor、講師へ通知。

**L-REC-003｜ユーザー管理・担当管理にアクセス可［権・B・自動e2e］**

#### 再鑑（REV）
**L-REV-001｜最終承認（再鑑）［正・A・自動e2e/pytest］**
- 前提：received。
- 手順：再鑑ログイン→参照画面で再鑑（最終承認）。
- 期待：→ admin_approved（完了）、以降編集不可、PDF/Excel出力可。

**L-REV-002｜完了後の差戻し［正・B・pytest］**
- 前提：admin_approved。

#### 管理者・管理責任者（MAS/CHF）

**L-MAS-002｜先月以前の未処理クローズ［正・B・自動e2e/pytest］**

- 期待：管理責任者だけが skip_parent_approval 設定や chief 招待を操作できる。

### 6.3 新システム（イスト勤怠レポート for EMPS）

#### 講師（TUT）
**N-TUT-001｜業務連絡表の作成（勤務/有給/欠勤を含む）［正・A・自動e2e］**
- 前提：経理が当該講師×学校の契約を登録済み（派遣先=学校が選択可）。
- 手順：報告書一覧→派遣先（学校）選択→明細入力（例：6/16 勤務 17:00-18:30 担当時限1・2 指導90分／6/20 有給／6/21 欠勤）→保存。
- 期待：勤務は時間集計に加算、有給/欠勤は取得回数/欠勤回数で集計、同一日重複不可、下書き保存。

**N-TUT-002｜担当時限の複数選択／時刻5分刻み／終了<開始ガード［異・B・自動e2e］**
- 期待：担当時限は1〜10の複数選択（"1・3"）。時刻は5分刻み。終了が開始より前は不可。

**N-TUT-003｜学校へ依頼（提出）［正・A・自動e2e/pytest］**
- 期待：→ awaiting_school、学校へ通知。skip設定済みなら awaiting_office へ直接。

**N-TUT-004｜差戻し後の再依頼［正・A・自動e2e/pytest］**
- 前提：returned_to_tutor。
- 期待：修正→再提出で awaiting_school へ。差戻しコメントが見える。

**N-TUT-005｜契約由来項目は読取専用［正・B・自動e2e］**
- 期待：お客様ID・弊社担当・所在地・要望連絡事項等は講師側で編集不可（契約管理の値）。

**N-TUT-006｜他講師の報告書は不可［権・A・pytest］**
- 期待：自分の担当のみ。直接アクセスは拒否。

#### 学校（SCH）
**N-SCH-001｜承認［正・A・自動e2e/pytest］**
- 前提：awaiting_school。
- 期待：→ awaiting_office。

**N-SCH-002｜差戻し（コメント必須）［異・A・自動e2e/pytest］**
- 期待：未入力不可、入力後 returned_to_tutor、講師へ通知。

**N-SCH-003｜学校確認スキップ時の挙動［正・B・pytest］**
- 前提：管理責任者が当該学校に skip 設定。
- 期待：提出が awaiting_school を飛ばし awaiting_office から開始。

#### 事務（OFF）
**N-OFF-001｜事務確認（承認）［正・A・自動e2e/pytest］**
- 前提：awaiting_office。
- 期待：→ awaiting_sales。

**N-OFF-002｜差戻し［正・A・pytest］**
- 期待：awaiting_office→returned_to_tutor（講師へ）。

**N-OFF-003｜月分超過の事前確認フロー［正・B・pytest］**
- 前提：担当業務の月分が契約の月分固定を超過。
- 期待：提出時 awaiting_office_precheck。事務承認で awaiting_school へ合流、事務差戻しで講師へ。

**N-OFF-101｜ユーザー削除/無効：自分自身は不可［権・A・自動e2e/pytest］（回帰: 40f73df）**
- 手順：ユーザー管理→自分の詳細→無効化/削除。
- 期待：ボタン無効（理由表示）、API直叩きでも409「自分自身は削除・無効化できません」。
