# テストケース仕様書

両システム（**指導報告・指導時間確認票** / **イスト勤怠レポート for EMPS**）の全ロール・全操作を、
人が行う操作パターンで網羅的にテストし、バグ・不具合を洗い出して客先導入可能な品質に仕上げるための仕様。

---

## 1. 本書の読み方

各テストは「**だれが／なにをする／こうなればOK**」で書いています。専門用語・内部コードは使いません。

| 列 | 意味 |
|---|---|
| 番号 | テスト識別番号（例：既-講-01＝既存システム・講師・1番） |
| だれが | 操作する人（ロール） |
| なにをする | 画面で行う操作 |
| こうなればOK（期待結果） | この通りになれば合格。違えば不具合 |
| 種別 | 正常＝想定どおりの操作／異常＝誤入力・エラー時／権限＝アクセス制御／境界＝限界値・特殊データ／表示＝画面の見え方／出力＝PDF・CSV・帳票／通知＝メール |
| 重要 | 高＝必須（主要業務・重大）／中＝重要／低＝周辺 |
| 実施 | 自動＝Playwright(e2e)・pytestで自動確認／手動＝人が目視確認 |

**判定の記入**：実施時に各行へ「OK／NG」と気づきを記録します（NGはバグ票へ）。

## 3. 業務の流れ（合格の前提となる正しい順序）

**既存システム**
下書き →（講師が提出）→ **保護者承認待ち** →（保護者が承認）→ **運営提出（受付待ち）** →（受付が受領）→ **受領済み** →（再鑑が最終承認）→ **最終承認済み（完了）**
- どの段階でも、コメント必須の差戻しで「講師へ差戻し」に戻せる。完了後は再鑑が「受付へ差戻し」可能。
- 先月以前の未処理は管理者が「クローズ」できる。保護者承認をスキップ設定した担当は、提出時に保護者を飛ばす。

**新システム(EMPS)**
下書き →（講師が学校へ依頼）→ **学校確認待ち** →（学校が承認）→ **事務確認待ち** →（事務が承認）→ **営業確認待ち** →（営業が承認）→ **最終承認済み（完了）**
- 差戻し：学校/事務からは「講師へ差戻し」、営業/完了後からは「事務へ差戻し」。
- 月の時間が契約の固定時間を超える報告は、提出時にまず「事務の事前確認」を通す。管理責任者の設定で学校確認を省略できる。

## 4. テスト環境とメールの安全

- 環境：開発環境（http://localhost:8000・8001、メール確認は MailHog http://localhost:8025）。
- 全アカウントのパスワードは `Passw0rd!`。
- ⚠️ **メール安全（厳守）**：開発環境は実Gmail送信設定のことがある。**提出・承認・差戻し・招待など「フロー操作」を伴うテストの前に、必ずメールを MailHog 経由（または送らない設定）に切替える**。実顧客・実アドレスへ送らない。テスト後は元へ戻す。
- 参照・ログイン・閲覧のみのテストはメールを送らないため安全。

## 6. 自動／手動の実装メモ

- **自動 e2e（Playwright・実画面/API）**：`e2e/tests/` — ログイン全ロール、直近改修の回帰（ブランド改称・指導報告作成・カード順・自分自身ガード）、承認フロー（既存・新の全遷移）。
- **自動 pytest（API・サービス層・実送信ゼロ）**：`backend/tests`・`new_backend/tests` — ワークフロー遷移、権限、削除/無効ガード、採番、出力生成（列幅・行数・空行除外）、メールのキュー投函。
- **手動（目視＋スクショ）**：PDF/帳票のレイアウト、PC/スマホ表示、メール文面、契約管理の反映、操作感。
- フロー系の自動実行は**メールを MailHog 経由（実送信ゼロ）にしてから**行う（§4）。

## 7. 実施記録（直近）

- e2e スイート：32件 + 承認フロー（既存/新）green を確認（メールは MailHog 捕捉＝実送信ゼロ→Gmail設定へ復元済み）。
- 既存フロー（講師→保護者→受付→再鑑）・新フロー（講師→学校→事務→営業）とも全遷移パスを実機確認。
- 確認できた仕様（バグではない）：既存＝当月1報告/承認後追加不可、新＝契約未登録は作成不可、営業は編集モーダルなし。
- 各テストの「OK/NG・気づき」は実施時に本表へ追記する。
