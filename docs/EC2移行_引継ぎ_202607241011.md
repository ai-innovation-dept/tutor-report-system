## 3. 移行の全体方針（推奨＝リフト&シフト）

**同じ `docker compose` 構成をそのまま EC2 で再現し、DB だけデータ移行する**のが最短・低リスク。アプリ側のコード変更はほぼ不要（`.env` の URL だけ更新）。

### EC2 サイジング（推奨）
| 項目 | 推奨 | 補足 |
|---|---|---|
| インスタンスタイプ | **t3.small（2GB）＋ swap 2GB** か **t3.medium（4GB）** | 現行2GBは**RAMだけ逼迫**気味（ピーク ~1.6–1.8GB）。余裕重視なら t3.medium |
| OS(AMI) | Ubuntu 22.04 or 24.04 LTS | 現行と揃えるなら 22.04 |
| ストレージ | gp3 **30GB** | DBは小さい（ダンプ数十KB〜）。60GBは過大 |
| ネットワーク | **Elastic IP** を割当 | DNS を向ける固定IP。停止中課金に注意（起動中は無料） |
| CPU課金 | t3 は**バースト**。定常CPUは低負荷なので Unlimited 不要 | |

### セキュリティグループ（inbound）
- **22 (SSH)**：可能なら**社内IPに限定**（0.0.0.0/0 は避ける）
- **80 / 443**：全許可（nginx がここで受ける）
- **5432 / 8025 / 8000 / 8001 は開けない**（外部公開しない。nginx が内部で 8000/8001 へプロキシ）

## 4. 移行手順（ランブック）

> ⚠ 実行系は Claudチャット側で最終確認しながら進める。ここは設計と要点。DB移行はサーバー間直結（チャットにダンプを貼らない）。

### Step 1. EC2 プロビジョニング
- 上記サイズで起動 → Elastic IP 割当 → セキュリティグループ設定 → SSH 接続確認。

### Step 2. Docker / Compose 導入
```bash
sudo apt-get update && sudo apt-get install -y docker.io docker-compose-plugin git
sudo systemctl enable --now docker
sudo usermod -aG docker ubuntu   # 再ログインで sudo なし docker 可
```

### Step 3. コード配置＋`.env` 転送
```bash
git clone <GitHubリポジトリURL> tutor-report-system
cd tutor-report-system
# .env は git 管理外。Lightsail の .env を scp でそのまま持ち込む（後で URL だけ書き替え）
#   ローカル経由: scp ubuntu@52.197.43.164:tutor-report-system/.env ./  → scp で EC2 へ
```

### Step 4. DB データ移行（Lightsail → EC2）
```bash
# --- Lightsail 側でダンプ ---
sudo docker compose exec -T db pg_dump -U postgres -d tutor > tutor_$(date +%Y%m%d_%H%M%S).sql
# --- そのファイルを EC2 へ転送（scp）---
# --- EC2 側：先に db だけ起動してから流し込む ---
docker compose up -d db
cat tutor_YYYYMMDD_HHMMSS.sql | docker compose exec -T db psql -U postgres -d tutor
```
> 代替：`postgres-data` ボリュームを丸ごと `tar` で移送してもよいが、**PG同一メジャー(16)前提**。`pg_dump/psql` の方が安全で推奨。

### Step 5. 起動＋マイグレーション確認
```bash
docker compose up -d --build
docker compose exec backend alembic upgrade head           # legacy（起動時にも自動実行）
docker compose exec new_backend python -m alembic upgrade head  # EMPS
docker compose exec -T db psql -U postgres -d tutor -c "SELECT count(*) FROM users;"  # データ疎通確認
```

### Step 6. DNS（`haken.net` サブドメイン → EC2 の Elastic IP）
- `kintai-yoyogi.haken.net` → 代々木進学会（:8000）
- `kintai-emps.haken.net` → EMPS（:8001）
- 既存の `*.haken.net` ワイルドカードを**個別Aレコードで上書き**して EIP を指す（第2レベルのラベル1つで統一する。入れ子は避ける）。

### Step 7. nginx リバースプロキシ＋HTTPS（Let's Encrypt）
- Host ヘッダで振り分け：`kintai-yoyogi.*` → `127.0.0.1:8000` / `kintai-emps.*` → `127.0.0.1:8001`。
- certbot でHTTPS化（2ドメイン個別、または `*.haken.net` ワイルドカードは DNS-01）。
- nginx もコンテナに載せて compose に追加するか、ホスト直インストールかは移行チャットで判断（既存compose追加が構成一元化で楽）。

### Step 8. `.env` の公開URL更新（★必須。メールリンクに直結）
```env
BASE_URL=https://kintai-yoyogi.haken.net
NEW_BASE_URL=https://kintai-emps.haken.net
```
→ `docker compose up -d --force-recreate` で反映。

### Step 9. 動作確認 → 切替 → 旧環境停止
- 両システムのログイン／報告書表示／PDF出力／（sandboxで）メールリンクが新URLになっているか確認。
- 問題なければ DNS 伝播を待って正式切替 → Lightsail は**しばらく残してから**停止/削除（ロールバック用）。

## 5. 移行で必ず気をつける点（このアプリ固有）

- **`.env` は2システム共有・DBも共有**。片方だけの設定にしない。`BASE_URL`(legacy)/`NEW_BASE_URL`(EMPS) は**別々のURL**。
- **メールリンクは `BASE_URL`/`NEW_BASE_URL` から生成**。更新漏れると招待・通知メールのリンクが旧IPのままになる（Step 8必須）。
- **メールは既定 `console`（送信オフ）**。移行検証中は送信させない。実配信は検証後に `bash mailmode.sh live`（`.env` の `MAIL_LIVE_*` 設定要）。**検証は sandbox 推奨**（実ユーザーに届かない）。
- **APScheduler はプロセス内**＝uvicorn を**単一ワーカー維持**。複数ワーカー/多重起動でリマインド重複送信になる。
- **MailHog は開発用**。本番でポート8025を外部公開しない（SGで塞ぐ）。将来 outbox の確認UIとして内部利用するなら nginx 配下＋Basic認証を検討。
- **DB認証情報の既定が `postgres/postgres`**。5432を外部公開しない前提だが、EC2移行を機に**強いパスワードへ変更**推奨（変更時は `DATABASE_URL` も同時更新）。
- **`JWT_SECRET`**：EC2 でも同値を使えばログイン状態の考え方は不変（値を変えると既存Cookieは無効＝再ログインになるだけで害はないが、意図せず変えない）。**本番用の十分に長い値**であることを確認。
- **バックアップ**：現行は自動バックアップ未設定（`backups/` に手動ダンプ1件のみ）。EC2移行を機に **cron で日次 `pg_dump` → 別ストレージ(S3)** を設定推奨。
- **タイムゾーン**：`TIMEZONE=Asia/Tokyo`（スケジューラ・締切計算に影響）。EC2 の `.env` でも維持。

## 6. まだ決めていない事項（移行チャット開始前に確認したい）

1. **EC2サイズ**：t3.small(2GB)+swap（低コスト）か t3.medium(4GB)（余裕）か。→ 既定は**t3.medium**を推奨。
2. **DB方式**：コンテナ内PostgreSQL維持（最短）か、この機に **RDS/マネージドDB** へ出すか。→ 既定は**コンテナ維持**（移行後にRDS化も可）。
3. **同時にHTTPS/ドメイン化するか**：純粋なリフト&シフト（IPのまま）を先に済ませ、後でドメイン化する二段構えか、一気通貫か。→ 既定は**一気通貫（Step 6–8まで実施）**。
4. **ダウンタイム許容度**：DBダンプ→復元の間の停止（数分想定）を許容できるか、無停止寄せが必要か。
5. **SSH鍵の運用**：EC2用キーペアの管理場所（Lightsailのキーとは別物）。
