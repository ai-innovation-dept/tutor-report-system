## 3. コンテナ構成（docker compose）

| service | image / build | ポート | 役割 | 永続化 |
|---|---|---|---|---|
| `db` | `postgres:16-alpine` | 5432 | 共有DB `tutor`（両システム） | named volume `postgres-data` |
| `mailhog` | `mailhog/mailhog:v1.0.1` | 8025 / 1025 | 開発用メール捕捉（本番は原則不使用） | なし |
| `backend` | `./backend`（Python3.11-slim） | 8000 | 代々木進学会（FastAPI＋Jinja2） | ソースはbindマウント |
| `new_backend` | `./new_backend`（Python3.11-slim） | 8001 | EMPS（FastAPI＋Jinja2） | 同上 |

- 各アプリは起動時 `alembic upgrade head` → `uvicorn ... --port 800X`（**単一ワーカー**）。
- スケジューラ **APScheduler をアプリ内プロセス**で毎日 09:00 JST 実行。→ **ワーカーを増やすと多重発火**するため単一ワーカー厳守。

## 5. データ・永続化

- **共有DB `tutor`**（`postgres:16-alpine`・コンテナ内・RDS未移行）。`DATABASE_URL` 一本のため将来RDS化が容易。
- 永続データは named volume `postgres-data`。**`docker compose down -v` は厳禁**（DB全消去）。
- `users` / `assignments` / `invitations` は**両システム共有**。EMPS専用は `work_*`。
- マイグレーションは legacy=`alembic_version` / EMPS=`work_alembic_version` で分離。共有テーブルは **legacy 側 Alembic が管理**。

## 6. 認証・セキュリティ設計

- 認証: **JWT を httpOnly Cookie（`access_token`）** で保持。パスワードは bcrypt。有効期限 `ACCESS_TOKEN_EXPIRE_HOURS`（既定8h）。
- 認可: RBAC（`backend/app/core/rbac.py`）。ロールは6種（代々木）/ EMPS 6種。
- ⚠ **通信の暗号化が未整備**（現状HTTP）。HTTPS化を最優先の残タスクとする。
- 秘密情報は `.env`（git管理外）。SSH秘密鍵 `.pem` は作業者PCローカル保管、リポジトリ・チャットに置かない。
- ⚠ DB認証情報の既定が `postgres/postgres`。外部非公開(5432閉)前提だが、運用中に強い値へ変更推奨。

## 7. メール送信

- **送信キュー(outbox)方式**: 即時送信せずアウトボックスへ投函→1通ずつ間隔送信（`MAIL_SEND_INTERVAL_SECONDS` 既定4秒）。2システム横断でPGアドバイザリロックにより直列化。
- `MAIL_BACKEND`: `console`（ログのみ＝**送信オフ**・現在の設定）/ `smtp`（実送信）。
- 送信経路（HOST/PORT/TLS/認証）は両系共通、送信元のみ `SMTP_FROM`（代々木）/`NEW_SMTP_FROM`（EMPS）で分離。
- 切替は `mailmode.sh sandbox|off|live`（`.env` の `MAIL_SANDBOX_*`/`MAIL_LIVE_*` を参照して書換＋再起動）。
- ⚠ **本番から実メールを送るのは `MAIL_BACKEND=smtp` かつ実在SMTP の両方が揃ったときのみ**。現状は `console` なので実メールは飛ばない。
