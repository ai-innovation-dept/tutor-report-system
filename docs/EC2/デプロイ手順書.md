## 🚀 いつものデプロイ（普段はこれだけ）

Lightsail時代と流れは同じ。**変わるのはサーバーへの“つなぎ方”だけ**（Lightsailのブラウザ SSH は使えない）。実行コマンドは同一。

| | Lightsail（今まで） | EC2（これから） |
|---|---|---|
| ① 依頼 | 人間 → Claude Code | 同じ |
| ② 開発＋`git push` | Claude Code | 同じ |
| ③ 反映 | AWSログイン→LightsailブラウザSSH→コマンド | **つなぎ方だけ変更（下記A/B）** → コマンドは同一 |

**③で打つコマンド（今までと同じ）**
```bash
cd ~/tutor-report-system && git pull && sudo docker compose up -d --build
```

### つなぎ方A：ブラウザ（Lightsailのブラウザ SSH に一番近い・おすすめ）
1. AWSコンソール → **EC2** → 「インスタンス」→ `i-0ce3a2e284f376401` を選択
2. 右上 **「接続」** → **「EC2 Instance Connect」** タブ → **「接続」**（鍵ファイル不要）
3. 開いたターミナルで上記コマンドを実行

### つなぎ方B：自分のPCから鍵SSH
```powershell
# Windows PowerShell（鍵は作業者PCローカル）
ssh -i C:\Users\<ユーザー>\KintaiApp\tutor-ec2-key.pem ubuntu@52.199.22.60
# つながったら
cd ~/tutor-report-system && git pull && sudo docker compose up -d --build
```
> 「鍵の権限が緩い」と拒否されたら一度だけ: `icacls "鍵のパス" /inheritance:r /grant:r "$($env:USERNAME):(R)"`

> 接続先IPは **52.197.43.164（旧Lightsail）→ 52.199.22.60（EC2）** に変更。DBスキーマ変更を含む改修でも、起動時に `alembic upgrade head` が自動実行されるので上記コマンドだけでOK（手動適用は §2）。

以下は、上記で足りない場合の詳細（初回セットアップ・特殊手順・切替・トラブル対応）。

## 1. 通常デプロイ（コード反映）

### 1-1. ローカル → GitHub
```bash
git add <関連ファイル>
git commit -m "…（管理番号・項目番号を含む）"
git push
```

### 1-2. EC2 で取り込み＆再構築
```bash
ssh -i tutor-ec2-key.pem ubuntu@52.199.22.60
cd ~/tutor-report-system
git pull
sudo docker compose up -d --build
```
- `up -d --build` は**コード反映とマイグレーションのみ**。ユーザー等のデータは消えない。
- 起動時に各アプリが `alembic upgrade head` を自動実行する。

### 1-3. 反映確認
```bash
sudo docker compose ps                 # 4サービスが Up
sudo docker compose logs backend --tail=50
sudo docker compose logs new_backend --tail=50
curl -s -o /dev/null -w "%{http_code}\n" http://localhost:8000/   # 200
curl -s -o /dev/null -w "%{http_code}\n" http://localhost:8001/   # 200
```

## 2. DBマイグレーション

通常は起動時に自動適用される。手動で行う場合：
```bash
sudo docker compose exec backend alembic upgrade head            # 代々木（共有テーブル含む）
sudo docker compose exec new_backend python -m alembic upgrade head  # EMPS（work_*）
sudo docker compose exec backend alembic current                 # 現在リビジョン確認
```

## 3. メール送信モードの切替

```bash
sudo bash mailmode.sh off       # 送信オフ（console・現在の既定）
sudo bash mailmode.sh sandbox   # 検証（Mailtrap等・実ユーザーに届かない）
sudo bash mailmode.sh live      # 実配信（.env の MAIL_LIVE_* が必要）
```
- ⚠ **本番切替が完了し `BASE_URL`/`NEW_BASE_URL` を正しいURLに更新してから `live`** にする（メールリンク不整合を防ぐ）。
- 現状の確認:
```bash
sudo docker compose exec backend python -c "from app.config import settings as s; print(s.mail_backend, s.smtp_host, s.smtp_port, s.smtp_tls, s.smtp_from)"
```

## 4. 一回限りの特殊手順（必要時のみ）

### 4-1. allowed_systems 正規化（分離リリース初回のみ・冪等）
```bash
sudo docker compose exec -T db pg_dump -U postgres -d tutor > backup_$(date +%Y%m%d_%H%M%S).sql
sudo docker compose exec backend python -m app.scripts.normalize_allowed_systems
```

### 4-2. 本番クリーン投入（破壊的・初回セットアップのみ）
> ⚠ **全データを削除**しサンプル6ユーザーのみにする。実ユーザー運用開始後は実行しない。
```bash
sudo docker compose exec -T db pg_dump -U postgres -d tutor > backup_$(date +%Y%m%d_%H%M%S).sql
sudo docker compose exec backend python -m app.scripts.seed_production --yes
sudo docker compose exec -T db psql -U postgres -d tutor -c "SELECT user_no,email,role FROM users ORDER BY user_no;"
```
`ENVIRONMENT=production` を `.env` に設定しておくこと（migration 0014 のガード有効化）。

## 5. 環境変数（`.env`）の変更

```bash
nano ~/tutor-report-system/.env          # 例: BASE_URL/NEW_BASE_URL、MAIL_* を編集
sudo docker compose up -d --force-recreate  # 反映（再ビルド不要な変更でも再作成で反映）
```
- `.env` は**両システム共有**。`BASE_URL`（代々木）/`NEW_BASE_URL`（EMPS）は**メールリンク直結**。

## 6. バックアップ・リストア

### 6-1. 手動バックアップ（現状の方法）
```bash
sudo docker compose exec -T db pg_dump -U postgres -d tutor > backup_$(date +%Y%m%d_%H%M%S).sql
# S3へ（IAMロールでキー不要）
aws s3 cp backup_*.sql s3://tutor-report-system-backup-nxtech2026/manual/
```
### 6-2. リストア
```bash
cat backup_YYYYMMDD_HHMMSS.sql | sudo docker compose exec -T db psql -U postgres -d tutor
```
> 日次自動バックアップ（cron）は**未実装（次タスク）**。実装時は `pg_dump`→`aws s3 cp`＋世代管理を cron 化する。

## 7. ロールバック

- **コード**: 直前へ戻す
```bash
cd ~/tutor-report-system && git log --oneline -5
git checkout <戻したいcommit> -- <ファイル>   # または git revert <commit>
sudo docker compose up -d --build
```
- **DB**: マイグレーションを1つ戻す（要注意）
```bash
sudo docker compose exec backend alembic downgrade -1
```
- 破壊的変更後は §6-2 のダンプからリストア。

## 8. 本番切替（Lightsail → EC2 カットオーバー）※未実施

現在は二重稼働（Lightsail=現行 / EC2=検証）。切替時の推奨順序：
1. **最終データ同期**: Lightsail で `pg_dump` → EC2 で `psql` 復元（二重稼働中に増えた差分を取り込む）。
   ```bash
   # Lightsail 側
   sudo docker compose exec -T db pg_dump -U postgres -d tutor > cutover_$(date +%Y%m%d_%H%M%S).sql
   # → scp で EC2 へ転送 → EC2 側で復元（必要ならDB初期化後）
   ```
2. **DNS切替**: `kintai-yoyogi.haken.net` / `kintai-emps.haken.net` を `52.199.22.60` へ（`*.haken.net` を個別Aで上書き）。
3. **nginx + HTTPS**: リバースプロキシ（Host振り分け 8000/8001）＋ Let's Encrypt。
4. **`.env` 更新**: `BASE_URL`/`NEW_BASE_URL` を新HTTPS URLへ → `up -d --force-recreate`。
5. **SG締め**: 8000/8001 を閉じ、22/80/443 のみに。
6. **メール実配信**: `mailmode.sh live`（必要時）。
7. **旧Lightsail**: しばらく残し（ロールバック用）、問題なければ停止/削除。

（詳細な設計は `docs/EC2移行_引継ぎ_202607241011.md` Step6–8）

## 9. トラブルシュート

| 症状 | 確認 |
|---|---|
| アプリが応答しない | `sudo docker compose ps` / `logs <svc> --tail=100` |
| メモリ逼迫 | `free -h`（swap 4GB利用状況）/ `docker stats` |
| ディスク不足 | `df -h` / `docker system prune`（未使用イメージ削除） |
| DB接続不可 | `logs db` / `docker compose exec db pg_isready -U postgres -d tutor` |
| メールが届かない | §3 の設定確認（`console` なら送信オフが正常） |

> ⚠ **本番メール事故防止**: 検証は必ず `MAIL_BACKEND=console` の pytest／`sandbox` で行う。実SMTPで承認フローを手動実行しない（送信キュー `mail_outbox`/`work_mail_outbox` の行で確認）。
